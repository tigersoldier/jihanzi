# ADR 0001: 使用追加日志作为数据源

**日期**：2026-06-28
**状态**：已采纳
**决策者**：项目架构设计

---

## 背景

记汉字是一个纯前端 PWA，数据存储在用户个人的 Google Drive 上。
多设备场景（手机 + 电脑）和多用户场景（父母双方各自登录）要求
数据能够无冲突合并。传统 CRUD 的"最后写入胜出"策略在离线 + 多设备
场景下会导致数据丢失和冲突。

## 决策

**使用追加日志 (Append-Only Log) 作为唯一数据源。**

- 每个用户操作产生一条不可变日志条目，带毫秒级时间戳
- 当前应用状态通过从快照起重放日志重建
- 日志条目超过 500 条时生成新快照，清理旧日志
- 多设备合并取日志并集——条目不可变，天然无冲突

## 后果

**优点**：

- 天然无冲突：条目追加不修改已有数据，合并只需取并集。不需要 CRDT、
  向量时钟等复杂冲突解决机制。
- 完整审计：每条操作都有时间戳记录，可追溯任意时间点的状态。
- 离线友好：离线时追加到本地 IndexedDB，联网后整批推送，不需要
  在断网期间做冲突检测。
- 简单可验证：状态重建是纯函数（log × snapshot → state），可单独
  测试。

**缺点**：

- 读时计算：每次读取状态都需要重放日志，计算量随日志量线性增长。
  通过快照机制缓解——只重放快照之后的新日志。
- 存储膨胀：不删除的历史操作会持续占用空间。通过快照 + 日志压缩
  机制缓解——超过 500 条日志自动生成新快照并清除旧条目。
- 不可修改：日志条目写入后不可更改，如需"撤回"一条操作，必须追加
  一条反向操作（例如 `add_char` → `remove_char`）。

## 备选方案

### 方案 B：传统可变更状态 + 最后写入胜出

直接存储和修改 JSON 对象（children、wordBooks），多设备同步时比较
修改时间戳，最后写入覆盖。

- **为什么不选**：在线/离线交替场景下，设备 A 离线修改了生字本，
  设备 B 离线做了评分，一旦 A 先上线推送再被 B 的后上线推送覆盖，
  生字本修改就丢失了。"最后写入胜出"需要全量读取 → 修改 → 全量
  写入，无法做字段级合并。

### 方案 C：CRDT (Conflict-Free Replicated Data Types)

使用 CRDT 库（如 Yjs、Automerge）处理状态同步。

- **为什么不选**：CRDT 的复杂性远超过本项目所需。本项目的操作天然
  可建模为不可变事件的追加（创建孩子、添加汉字、评分），不需要
  CRDT 的字符级/字段级合并能力。CRDT 还引入了额外的元数据
  开销（向量时钟），对于以日志条目为自然单位的中文教学场景来说
  过于重型。
