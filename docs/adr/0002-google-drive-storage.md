# ADR 0002: 使用 Google Drive 作为云端存储

**日期**：2026-06-28
**状态**：已采纳
**决策者**：项目架构设计

---

## 背景

记汉字需要在多设备间同步数据，并支持家庭成员（父母双方）共享同一份
学习数据。传统的后端服务方案需要服务器、数据库、用户认证系统，对
一个面向个人的小型工具来说运维成本和费用都不合理。

## 决策

**使用用户个人的 Google Drive 作为唯一云端存储，完全无后端。**

- 前端通过 Google Identity Services 做纯前端 OAuth 认证
- 通过 Google Drive API v3 直接读写用户 Drive 上的文件
- 文件以 JSON / JSONL 格式存储，组织在 `记汉字/` 顶层文件夹下
- 多用户共享通过 Drive 原生共享机制实现——父母一方将文件夹共享
  给配偶，子文件自动继承权限

## 后果

**优点**：

- 零服务器成本：不需要服务器、数据库、运维。GitHub Pages 托管
  静态资源，Google Drive 托管用户数据，全部免费。
- 用户掌控数据：数据存储在用户自己的 Google 账号下，不经过第三方
  服务器。用户随时可以在 Drive 中查看、导出、删除数据。
- 安全模型依托 Google：OAuth 认证和文件权限管理由 Google 负责，
  不需要自己实现密码存储、会话管理、访问控制。
- 天然的共享机制：无需开发"家庭组"功能，用户直接使用 Drive
  原生的文件夹共享即可实现多家长协作。

**缺点**：

- 依赖 Google 生态：用户必须有 Google 账号。非 Google 用户无法
  使用云端同步功能（但 app 仍可在离线模式下本地使用）。
- API 配额限制：Google Drive API 有每日请求配额（免费额度为
  每 100 秒 100 次请求，每日约 86,400 次），远超本应用的预期
  使用量（用户每天几十次操作 + 每 5 分钟一次后台同步），但仍
  需注意不滥用。
- 启动时加载：首次加载需要从 Drive 拉取全部数据。对于正常使用
  场景（几百到几千条日志 + 快照），加载时间在几百毫秒内，不影响体验。
- Google API 脚本加载：需要从 Google CDN 加载 `gapi` 和 `gis`
  两个外部脚本，在无网络时这些脚本不可用——此时 app 自动降级为
  离线模式。

## 备选方案

### 方案 B：Firebase / Supabase BaaS

使用 Firebase Firestore 或 Supabase 作为后端即服务。

- **为什么不选**：引入了第三方服务依赖和成本。免费额度有限，
  超出后需要付费。且仍然需要实现用户认证和多用户共享逻辑。
  相比之下，Google Drive 完全免费且用户已有账号。

### 方案 C：纯本地存储，手动导出导入

数据只存 IndexedDB，用户在设备间手动导出/导入 JSON 文件。

- **为什么不选**：手动操作繁琐，容易忘记备份/同步，导致数据
  不一致。失去了 PWA 的便利性。Google Drive 方案在 OAuth 后
  自动同步，无需用户主动操作。

### 方案 D：搭建自有后端

使用 Node.js + PostgreSQL，部署在 VPS 或云服务上。

- **为什么不选**：对于面向个人的小型工具，运维成本过高。需要
  管理服务器安全、数据库备份、用户认证、API 限流等，与"简单
  工具"的定位不符。
